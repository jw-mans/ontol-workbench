# Дизайн: импорт связей (`with relationships`)

Рабочий проектный файл. Описывает фичу «импорт отношений между импортируемыми
сущностями». По нему будем реализовывать. Статус: **черновик, обсуждается.**

## 1. Проблема (мотивация)

Сейчас при импорте парсер пропускает безымянные определения
([parser.py](../src/ontol/parser.py), цикл слияния, `if definition.name is None: continue`).
Связи (`Relationship`) по умолчанию **безымянны** (`name = None` в [oast.py](../src/ontol/oast.py)),
поэтому блок `hierarchy:` из импортируемого файла **не переносится**:

- `import { … } from 'mod.ontol'` тянет типы и функции, но не иерархию из `mod.ontol`;
- проявление: в `examples/multifile_demo/full.ontol` рёбра наследования/агрегации
  не появляются на диаграмме, хотя типы и функции — да.

Обходной путь сегодня: объявлять `hierarchy:` прямо в entry-файле.

## 2. Идея (из обсуждения)

- Связи остаются **безымянными** — модель не меняем.
- Импорт связей делается **по желанию**, через ключевое слово `with relationships`.
- Импортируются только те связи исходного модуля, у которых **оба конца**
  (`parent` и все `children`) попали в число импортированных сущностей. Иначе
  притащили бы связь на тип, которого в целевом файле нет.

## 3. Синтаксис

```ontol
import { set, element } from 'base.ontol' with relationships
import * from 'base.ontol' with relationships
```

- `with relationships` — необязательный хвост у обеих форм импорта.
- Без него — поведение как сейчас (связи не тянутся). Обратная совместимость сохраняется.

## 4. Семантика

Пусть `Imported` — множество имён сущностей, фактически добавленных этим импортом
(с учётом алиасов — см. §6). При `with relationships` для каждой связи `r` из
`ontology.hierarchy` исходного модуля:

- взять `r` тогда и только тогда, когда `r.parent.name ∈ Imported` **и**
  `child.name ∈ Imported` для всех `child` из `r.children`;
- иначе — пропустить.

Для `import *` множество `Imported` = все сущности модуля, поэтому проходят все
связи модуля.

## 5. Дедупликация связей

Связи безымянные, поэтому проверка «по имени» не работает. Дедуп — **по значению**
(`dataclass __eq__`, как сделали для термов при diamond-импорте):

- перед добавлением связи проверяем, нет ли уже равной по содержимому в
  `self.__ontology.hierarchy`; если есть — пропускаем.

Это заодно корректно обрабатывает diamond (одна связь, пришедшая по двум путям).

## 6. Тонкие места

### 6.1 Алиасы
`import { set as collection } with relationships`:
- если у связи конец `set`, а импортировали его как `collection`, имя конца надо
  **переименовать** `set → collection` в копии связи;
- если какой-то конец связи вообще не импортирован (например `subset`) — связь
  пропускается (правило §4).
- Решение: строим карту `original_name → final_name` по импортируемым сущностям и
  применяем её к копии связи. Работаем с **копией** (`copy.deepcopy`), чтобы не
  портить исходный модуль.

### 6.2 `import *`
Все сущности импортированы → все связи проходят; алиасов нет. Тривиально.

### 6.3 Конечные термы связи
`Relationship.parent` / `children` — это `Term`-объекты. После переноса в целевую
онтологию у нас уже есть одноимённые термы (их импортировали как сущности). Для
рендера PlantUML связи сопоставляются по **имени**, поэтому достаточно, чтобы имена
совпадали; перепривязывать объекты не обязательно (но можно — на будущее).

### 6.4 Циклы импорта
Не затрагивается: детекция циклов работает на уровне файлов до слияния определений.

## 8. Тесты (что добавить)

- `import *` … `with relationships`: иерархия модуля переносится.
- `import { A, B } with relationships`: переносится только связь, у которой оба
  конца импортированы; связь с неимпортированным концом — нет.
- без `with relationships`: связи по-прежнему НЕ переносятся (регрессия дефолта).
- diamond: одна и та же связь по двум путям → одна копия (дедуп по значению).
- алиас: `import { set as collection } with relationships` → конец связи
  переименован в `collection`.
- (опц.) обновить `examples/multifile_demo/full.ontol`, чтобы рёбра появлялись
  через `with relationships`, и убрать обходной локальный `hierarchy:`.

  алиасированным концом просто пропускается)?
