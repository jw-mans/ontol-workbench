# ONTOL V1 — отчёт о работе

Отчёт описывает, как устроен ONTOL V1, и фиксирует доработки этой недели
(личный кабинет с модульными проектами). План — в [../../../V1_PLAN.md](../../../V1_PLAN.md),
грамматика — в [GRAMMAR.md](GRAMMAR.md), сборка/запуск — в [SETUP.md](SETUP.md).

## 1. Назначение

ONTOL V1 — DSL и инструмент для описания онтологий и их визуализации в виде
диаграмм классов UML. По текстовому `.ontol`-файлу строятся: JSON-представление,
исходник PlantUML и отрисованная диаграмма (PNG). Ключевая особенность V1 —
**модульность**: онтологию можно разнести по нескольким файлам, импортирующим
друг друга.

## 2. Конвейер обработки

```
.ontol  ──Lexer──►  токены  ──Parser──►  Ontology (AST)  ──┬─► JSONSerializer  ─► .json
                                                           ├─► PlantUML        ─► .puml ─► .png
                                                           └─► Retranslator    ─► .ontol (debug)
```

Компоненты ([src/ontol/](../src/ontol/)):

| Файл | Роль |
|------|------|
| `parser.py` | лексер + парсер (на `sly`), разрешение импортов, построение `Ontology`. |
| `oast.py` | модель онтологии: `Ontology`, `Term`, `Function`, `Relationship`, `Figure`, перечисления. |
| `serializer.py` | сериализация `Ontology` → JSON. |
| `plantuml.py` | генерация исходника PlantUML и рендер PNG (через сервер PlantUML). |
| `retranslator.py` | обратная трансляция `Ontology` → `.ontol` (для отладки, флаг `--debug`). |
| `ai.py` | опциональная генерация связей LLM (`--gen-hierarchy`); langchain грузится лениво. |
| `cli.py` | интерфейс командной строки. |
| `project.py` | **новое:** слой проектов для ЛК (см. §5). |
| `auth.py` | **новое:** учётные записи пользователей (см. §5). |

## 3. Модель языка

Онтология состоит из секций (см. полную грамматику в [GRAMMAR.md](GRAMMAR.md)):

- **метаданные** — `version`, `title`, `author`, `description`, `type`, `date`;
- **`types:`** — термы (классы) с меткой, описанием и атрибутами оформления;
- **`functions:`** — функции (операции) с типизированными аргументами и результатом;
- **`hierarchy:`** — отношения между термами; тип отношения — один из
  `dependence`, `association`, `directAssociation`, `inheritance`,
  `implementation`, `aggregation`, `composition`;
- **`figure '...':`** — именованные подсхемы (подмножества определений для отдельной диаграммы).

Семантические проверки (уникальность имён, ссылки на существующие термы,
допустимые ключи атрибутов и значения) выполняются парсером и сопровождаются
сообщениями с указанием файла, строки и позиции.

## 4. Импорты (модульность)

Импорт реализован на уровне языка:

```ontol
import { set, element as elem } from 'base.ontol'   (* выборочно, с алиасами *)
import * from 'base.ontol'                           (* всё *)
```

- путь резолвится относительно папки импортирующего файла; поддерживается URL;
- импортированные определения добавляются в текущую онтологию (с учётом алиасов);
- **защита от циклов:** парсер ведёт цепочку импортируемых файлов и при повторном
  входе выдаёт ошибку вида `circular import detected: a.ontol -> b.ontol -> a.ontol`
  вместо аварийного `RecursionError`. Diamond-импорт (один файл по двум разным
  путям) циклом не считается.
- **дедупликация при diamond-импорте:** если одно и то же определение приходит по
  двум путям, повтор молча отбрасывается (сравнение по значению); ошибка
  `already declared` остаётся только при реальном конфликте — разных определениях
  с одинаковым именем.

## 5. Личный кабинет (доработка недели)

Добавлены слой проектов [src/ontol/project.py](../src/ontol/project.py),
слой авторизации [src/ontol/auth.py](../src/ontol/auth.py) и
веб-интерфейс [deploy/app.py](../deploy/app.py).

**Понятия:**
- **проект** — директория с несколькими `.ontol`-файлами;
- файлы в пределах проекта импортируют друг друга; т.к. импорт резолвится по
  реальным путям на диске, для сборки достаточно отрендерить выбранную точку входа.

**API слоя проектов:**
- `Project` — операции с файлами проекта (`list_files`, `read_file`, `write_file`,
  `delete_file`) с защитой от выхода за пределы директории (path traversal);
- `ProjectStore` — управление проектами как поддиректориями (`create`, `list_projects`,
  `get`, `delete`, `rename`, `exists`);
- `render_project(project, entry, output_dir)` — рендер **in-process**: парсит точку
  входа (импорты разрешаются по папке проекта) и формирует JSON + `.puml` + PNG,
  возвращая `RenderResult` (пути, предупреждения, ошибка). PNG — best-effort
  (нужен сервер PlantUML); при недоступности JSON/`.puml` всё равно формируются.

**Авторизация и привязка проектов к пользователям:**
- `UserStore` ([auth.py](../src/ontol/auth.py)) — регистрация и вход; пароли
  хранятся как соль + PBKDF2-HMAC-SHA256 (240k итераций) в JSON-файле, сверка
  через `hmac.compare_digest`; имена пользователей валидируются (безопасны как
  имя каталога). Реализовано на стандартной библиотеке, без новых зависимостей;
- проекты приватны для каждого пользователя: ЛК работает с персональным
  `ProjectStore` в `<projects_dir>/<username>` — пользователи не видят чужих проектов;
- путь к файлу пользователей и каталогу проектов настраивается через
  `ONTOL_USERS_FILE` и `ONTOL_PROJECTS_DIR`.

**Веб-ЛК (Streamlit):**
- экран входа / регистрации; в сайдбаре — текущий пользователь и кнопка «Выйти»;
- слева — список проектов (создать / переименовать / удалить) и кнопка «Загрузить демо»;
- редактирование файлов по вкладкам, добавление/удаление файлов, выбор точки входа;
- редактор с подсветкой синтаксиса и нумерацией строк (Ace через `streamlit-ace`,
  с откатом на обычное текстовое поле, если компонент не установлен) и
  **автосохранением** изменений на диск (кнопка «Сохранить» больше не нужна);
- «Собрать» → диаграмма + JSON + `.puml` + скачивание zip + список предупреждений;
- рендер выполняется в процессе приложения (без вызова внешнего бинарника).

## 6. Тестирование и доказательство работоспособности

- Юнит-тесты: `tests/` — **77 passed** (53 исходных, 13 по слою проектов,
  циклам импорта и diamond-дедупликации [tests/test_project.py](../tests/test_project.py),
  и 11 по авторизации [tests/test_auth.py](../tests/test_auth.py)).
- Демо-проект [examples/multifile_demo/](../examples/multifile_demo/): `main.ontol`
  импортирует `set`, `element` из `base.ontol` и строит корректную диаграмму
  (Subset ▷ Set, Element ◇* Set).
- Веб-ЛК проверен через `streamlit.testing.AppTest`: сценарий «холодный старт →
  экран входа → регистрация → загрузка демо → сборка → выход → повторный вход»
  проходит без исключений; проекты создаются в персональном каталоге пользователя.

## 7. Что сделано на этой неделе (итог)

1. Разведены зависимости: langchain стал опциональным, ядро/веб/тесты работают без него
   (`requirements-core.txt`, `requirements-ai.txt`).
2. Добавлен слой проектов (`Project`, `ProjectStore`, `render_project`) с тестами.
3. Реализована защита от циклических импортов с внятным сообщением и тестами,
   а также дедупликация одинаковых определений при diamond-импорте.
4. `deploy/app.py` переписан в многофайловый ЛК с проектами и вкладками;
   добавлены подсветка синтаксиса (Ace) и автосохранение файлов.
5. Добавлена авторизация пользователей (`UserStore`) и привязка проектов к
   пользователям (персональный `ProjectStore` на каждого), с тестами.
6. Подготовлены артефакты: грамматика в РБНФ ([GRAMMAR.md](GRAMMAR.md)) и данный отчёт.
